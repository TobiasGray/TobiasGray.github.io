Microbit

When deciding what to code, I ran into a major issue. Since the microbit is a 5x5 grid, I had limited space to make a functionally difficult game. Additionally, only flashing red LEDs and buttons could interact with the player. When brainstorming ideas that fit this criteria, I looked for inspiration in arcade and 1 2 3 4 player games. I decided to code my first ever Space Invaders Game. The final product had 120+ lines of code, so let's go over the challenges, successes, and more itinerary items I had.

STEP 1

I started with the player hitbox. My "ultra 4k" invader jet for the player included 3-4 level 7 LEDs in a triangle shape, depending on whether or not the player was on the edge. It was easy enough to incorporate movement from buttons through my if commands and functions. I quickly moved onto player bullets. These were a bit more tricky, as I needed to create a Class to contain multiple functions for the bullets, including creating them, moving them, checking if they made contact, and finally destroying them. Classes take more time to build than regular functions, as you need to construct self parameters and often require seperate documents to track these. This class allowed me to sync my player and bullet hitbox, as well as create a unique type of variable that could be globally used for these bullets.

STEP 2

The idea I had built in my head included enemy planes with bullets that could interact with player bullets. Unfortunately there weren't enough rows on the microbit to integrate this. I looked to the next best thing, which was dot enemies with light dot bullets. However, as time was running thin I decided to focus on adding basic functionality to the game before delving into details. I created a new class for enemies, and later decided there was only going to be one enemy at a time (though I seriously considered it capable and realistic to generate more). Enemies took a long time, as for a while my bullets were not interacting correctly with them. After a lot of coding, the enemies could die. However, I had no difficulty integrated into it! Currently, it was unfeasable to die or lose with the player character. Without enemy bullets, how could I integrate this?

STEP 3

The short answer? A timer. I incorporated two simple variables to add a losing option for the player: timer and score. The timer would continuosly tick down from 20 seconds, and the score would increase until they both hit a certain amount. For the time, this was 0 seconds. For the score, it was a number between 5 and 10 enemies, depending on what I thought was fair. Whichever happened first would stop the player and inform them the result. Simple, yet the principle is integral to many games played today.A goal I really wanted to implement was randomized movement for enemies so I called the random library. A challenge I faced at this point was the speed of the enemy. Every tick, bullets were updated and so was the player. Every one of these ticks was also updating the enemy, making them virtually impossible to track. A big part of the development process was using this challenge to implement a tracking system for the ticks that had already passed. Using a basic reminder function, I could make the enemy move twice, three times, four times as slow with the change of a singular counter. And so, I achieved the goal I set of having a fun game that could so easily be expanded.

Additional Document

For this project, I used one of my favorite coding organization tactics: A completely new document! I used this document for functions and classes. Here is everything this secondary document pertained:

    1. Drawing the player on the left, right, or anywhere in the middle
    2. Classifying and creating a class for bullets, including drawing them, initializing them, moving them, and destroying them
    3. Creating a class for the enemy, including initializing them, checking if they were drawn, updating them, randomizing their location and movement, and checking if the player should be awarded a point
    4. Checking if the player won! Adding music, and ending the game. This also stored the actual score variable
